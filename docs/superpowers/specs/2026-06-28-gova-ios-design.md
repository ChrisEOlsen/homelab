# gova-ios Template — Design Spec
**Date:** 2026-06-28
**Status:** Approved

---

## Overview

`gova-ios` is a standalone template repository that translates a web application built with
the `gova-monolith` stack into a native iOS application using SwiftUI. It is the first of two
mobile repos (`gova-ios`, `gova-android`) that form the mobile tier of the GOVA trifecta.

**The core premise:** the Go JSON API built by gova-monolith is already mobile-ready.
gova-ios only builds the iOS client. The backend is shared and untouched — except for one
additive change: a token-based auth endpoint is added alongside the existing cookie auth
so mobile clients can authenticate without browser session mechanics.

---

## What Is and Is Not Built

**In scope:**
- The `gova-ios` template repo (all config, commands, CLAUDE.md, SEED.md template, install script)
- The starter Xcode project (`ios/GovaApp/`) with pre-committed infrastructure files
- The `/build` Claude command that drives the full translation workflow
- The `install-claude.sh` setup script
- The design of `scaffold_mobile_auth` (implementation lives in gova-monolith)
- The design of `/export:mobile` (implementation lives in gova-monolith)

**Out of scope (this spec):**
- `gova-android` — separate repo, separate spec
- App Store deployment
- Push notifications, in-app purchases, background fetch
- `scaffold_mobile_auth` implementation in gova-monolith (separate task)
- `/export:mobile` command implementation in gova-monolith (separate task)

## Repo Structure

```
gova-ios/
├── .claude/
│   ├── commands/
│   │   └── build.md              ← /build command
│   └── settings.json             ← Claude permissions
├── .gitignore
├── .mcp.json                     ← optional gova-builder wire-up (generated by install script)

├── SEED.md                       ← app spec template (populated by /export:mobile)
├── install-claude.sh             ← setup script
└── ios/
    ├── GovaApp.xcodeproj/        ← pre-committed Xcode project
    └── GovaApp/
        ├── GovaAppApp.swift      ← app entry point (pre-committed, do not modify)
        ├── ContentView.swift     ← root NavigationStack shell (pre-committed)
        ├── Config.plist          ← API_BASE_URL (set by install script)
        ├── Lib/
        │   ├── APIClient.swift   ← shared HTTP client (pre-committed, do not modify)
        │   └── AuthManager.swift ← Keychain token store (pre-committed, do not modify)
        ├── Models/               ← Claude generates Swift structs here
        ├── Views/                ← Claude generates SwiftUI Views here
        └── ViewModels/           ← Claude generates ViewModels here
```

`Models/`, `Views/`, and `ViewModels/` are added to the Xcode project as **folder references**
(not groups), so Claude can add `.swift` files to them without touching `project.pbxproj`.

The SEED.md has two parts. The first is filled by the developer; the second is auto-populated
by `/export:mobile` in gova-monolith and must not be edited manually.

```markdown
# iOS App Specification

## App Name
[Your app name — used as the Xcode scheme and bundle display name]

## API Base URL
http://localhost:8080

## Design Notes
[Any iOS-specific UX notes. Otherwise follows CLAUDE.md defaults.]

---
## Generated Context
> Auto-populated by /export:mobile in gova-monolith. Do not edit manually.

### Screens
- items: GET /api/items, POST /api/items_create, DELETE /api/items/:id
  Fields: name (String), status (String)

### Auth
- Required: yes / no
- POST /api/auth/login_token   ← mobile token endpoint
- GET  /api/auth/me
- POST /api/auth/logout

### Notes
[observations about the web app's UX flows]
```

## CLAUDE.md — Full Content

```markdown
# Claude Code Context: GOVA iOS

You are the iOS Lead for a GOVA iOS app. Your job is to faithfully translate a web
application built with the gova-monolith stack into a native SwiftUI app. The Go
backend and JSON API are shared — you are only building the iOS client. The web app's
SEED.md (populated by /export:mobile) is the source of truth for screens, data models,
and endpoints.

---

## Translation Guide

### Step 1 — Read the SEED.md Generated Context
- Identify every screen (one per JS module in the web app)
- Identify every data model and its fields
- Identify which endpoints exist and their HTTP methods
- Note whether auth is required

### Step 2 — Extend the Go API for mobile auth (if auth is required)
- Web auth uses signed cookies — mobile cannot use these
- Call the gova-builder MCP tool scaffold_mobile_auth to add token endpoints
  alongside the existing cookie auth (the web auth system is untouched)
- This tool is idempotent — safe to call even if Android has already called it
- Mobile uses: POST /api/auth/login_token → returns { token, user }
- Store token in Keychain via AuthManager — never UserDefaults

### Step 3 — Define Swift models
- One Swift struct per data model in SEED.md
- Write to Models/ModelName.swift
- Field type mapping:
  - string  → String
  - int     → Int
  - boolean → Bool
  - float   → Double
  - created_at → Date (use ISO8601DateFormatter in APIClient)
- All models conform to Codable and Identifiable (id: Int)

### Step 4 — Build screens (one per JS module in the web app)
- Each screen = one View file (Views/NameView.swift) + one ViewModel (ViewModels/NameViewModel.swift)
- ViewModel owns all fetch logic and @Published state
- View is pure rendering — no network calls, no business logic
- Follow the web-to-iOS pattern mapping table below

### Step 5 — Wire navigation
- Use NavigationStack as the root container (already in ContentView.swift)
- List screens are top-level destinations
- Create/edit forms open as .sheet(isPresented:)
- Auth gate: if no token in AuthManager, push LoginView on appear

### Step 6 — Polish and verify
- Every network call must show a loading state and an error state in the UI
- Run: swift build (or xcodebuild) to confirm no compile errors
- Check: no force unwraps, no raw URLSession, no tokens in UserDefaults

---

## Web-to-iOS Pattern Mapping

| Web (JS module pattern)                    | iOS (SwiftUI equivalent)                              |
|--------------------------------------------|-------------------------------------------------------|
| loadList() → fetch → renderList()          | ViewModel.load() → @Published var items → List/ForEach|
| add_js_form creation form                  | .sheet(isPresented:) { Form { TextField... } }        |
| del('/api/x/:id') delete                   | .onDelete modifier on ForEach                         |
| requireAuth() at module top                | .onAppear { if !auth.isLoggedIn { nav to login } }    |
| api.js get(path)                           | APIClient.shared.get(path:)                           |
| api.js post(path, body)                    | APIClient.shared.post(path:body:)                     |
| api.js del(path)                           | APIClient.shared.delete(path:)                        |
| element.textContent = item.name            | Text(item.name)                                       |
| res.error ?? 'Something went wrong.'       | Text(errorMessage).foregroundStyle(.red)              |

---

## Architecture Rules (Non-Negotiable)

1. **MVVM always.** Never put fetch logic in a View. ViewModels only.
2. **APIClient always.** Never call URLSession directly. Use APIClient.shared.
3. **Keychain always.** Never store tokens in UserDefaults. AuthManager handles this.
4. **No force unwrap.** Never use ! in production code. Use guard let or optional chaining.
5. **@MainActor on ViewModels.** UI updates must happen on the main thread.
6. **async/await only.** No completion handler callbacks.
7. **NavigationStack.** Never use the deprecated NavigationView.
8. **Error states required.** Every API call must have a visible error state in the UI.
9. **Config.plist for URLs.** Never hardcode the API base URL. Read from Config.plist.

---

## Infrastructure Files (Pre-committed — Do Not Regenerate)

These files exist in ios/GovaApp/Lib/ and are the iOS equivalents of api.js and auth.js.
Never overwrite them. Import and use them.

- APIClient.swift — typed async/await HTTP client, reads base URL from Config.plist,
  injects Authorization: Bearer header automatically if a token exists
- AuthManager.swift — @MainActor ObservableObject, Keychain-backed token storage,
  exposes isLoggedIn, login(token:user:), logout()

---

## MCP Tools Available

The gova-builder MCP (from the connected gova-monolith instance) provides:

| Tool                  | When to use                                              |
|-----------------------|----------------------------------------------------------|
| inspect_app           | Read current web app models, handlers, routes            |
| scaffold_mobile_auth  | Add token auth endpoints to the Go API (idempotent)      |

All iOS file creation (Swift models, views, viewmodels) is done directly by Claude
using its file tools — no MCP tools are used for iOS file generation.
```

## /build Command Workflow

File: `.claude/commands/build.md`

```
Step 1 — Validate
  Read SEED.md. If the Generated Context block is empty or contains only placeholder text, STOP:
  > "The Generated Context section of SEED.md is empty. Run /export:mobile in your
  >  gova-monolith project first, then paste the output into SEED.md."

Step 2 — Brainstorm
  Use superpowers:brainstorming with SEED.md as input.
  Confirm: screen list, nav flow, auth requirement, any iOS-specific UX notes from the
  developer. Wait for developer approval before proceeding.

Step 3 — Write implementation plan
  Use superpowers:writing-plans.
  Plan order:
    1. scaffold_mobile_auth (if auth required) — idempotent, safe to run first
    2. Swift models (one per data model in SEED.md)
    3. Infrastructure validation (verify APIClient, AuthManager present)
    4. Auth screen (LoginView + LoginViewModel) if auth required
    5. One task per screen: ViewModel → View
    6. Navigation wiring in ContentView.swift
    7. Compile check (swift build)

Step 4 — Create feature branch
  Use superpowers:using-git-worktrees.
  Branch name derived from app name in SEED.md: "Task Manager" → build/task-manager

Step 5 — Implement
  Use superpowers:subagent-driven-development.
  Each subagent must check the CLAUDE.md translation guide before starting a screen.
  Infrastructure files (APIClient, AuthManager) are pre-committed — never regenerate.

Step 6 — Verify
  Run: xcodebuild -scheme GovaApp -sdk iphonesimulator build
  Check: every screen has a loading + error state, auth gate wired if required,
  no force unwraps (!), no raw URLSession, no tokens in UserDefaults.
  Use superpowers:verification-before-completion.

Step 7 — Done
  Report:
  > "Build complete."
  > "Open ios/GovaApp.xcodeproj in Xcode and run on simulator."
  > "Branch: build/[app-name]"
```

## install-claude.sh

**Prerequisites checked:** `git`, `xcodebuild` (verifies Xcode command line tools)

**Steps:**
1. Check git and xcodebuild present
2. Register superpowers plugin in `~/.claude/settings.json`
3. Prompt for API base URL (default: `http://localhost:8080`) → write to `ios/GovaApp/Config.plist`
4. Prompt for gova-monolith container name (optional) → generate `.mcp.json`
   - If skipped: write an empty `.mcp.json` with a comment instructing the developer to fill it in
5. Print next steps: fill SEED.md, run `/build`

**Does not:** touch Docker, Stripe, Gemini, OpenCode, SESSION_SECRET, or any `.env`.

## Swift Infrastructure — APIClient.swift

```
- Reads API_BASE_URL from Config.plist at init
- All requests inject Authorization: Bearer <token> header if AuthManager has a token
- Methods:
    get<T: Decodable>(path: String) async throws -> T
    post<T: Decodable>(path: String, body: Encodable) async throws -> T
    delete(path: String) async throws
- Decodes JSON with JSONDecoder using .iso8601 date strategy
- Throws APIError (network, decode, server(statusCode, message)) — never returns optionals
```

## Swift Infrastructure — AuthManager.swift

```
- @MainActor final class AuthManager: ObservableObject
- @Published var isLoggedIn: Bool (computed from Keychain token presence)
- @Published var currentUser: User? 
- func login(token: String, user: User) — saves token to Keychain, updates state
- func logout() — clears Keychain entry, resets state
- Keychain key: "gova.auth.token" (namespaced to avoid collisions)
- Injected as @EnvironmentObject from GovaAppApp.swift root
```

## scaffold_mobile_auth — Design Constraints for gova-monolith

When this tool is implemented in the gova-monolith builder, it must be **idempotent**:

- SQL: `CREATE TABLE IF NOT EXISTS mobile_tokens (...)` — safe to run twice
- Handler file: check if `handlers/mobile_auth.go` exists → skip with message if so
- Route check: scan `main.go` for `/api/auth/login_token` before appending — skip if found
- Output: clearly states "already scaffolded" vs "scaffolded fresh" so the developer knows state

Endpoints added to the Go API:
- `POST /api/auth/login_token` — validates credentials, returns `{ token, user }`, stores token hash in `mobile_tokens` table
- `DELETE /api/auth/logout_token` — invalidates the token (mobile logout)
- `GET /api/auth/me` — already exists from `scaffold_auth`, works with bearer token too

---

## /export:mobile — Design Constraints for gova-monolith

When this command is implemented in gova-monolith, it must produce:

1. A complete "Generated Context" block ready to paste into the mobile SEED.md
2. One screen entry per JS module in `static/js/` (excluding lib/)
3. Field names and types extracted from the corresponding Go model
4. All API routes from `main.go` relevant to each screen
5. Auth requirement inferred from `requireAuth()` presence in JS modules
6. Saved as a file (e.g. `mobile-seed-context.md`) at the repo root for easy copying

---

## Out of Scope

- `gova-android` — separate repo and spec
- App Store submission, signing, provisioning profiles
- Deep links, push notifications, background fetch, in-app purchases
- SwiftData or CoreData offline caching (API-first only)
- Unit or UI tests
